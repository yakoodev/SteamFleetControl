# Техническое задание

## Проект
Self-hosted сервис для централизованного управления несколькими собственными Steam-аккаунтами.

## 1. Цель проекта
Разработать self-hosted веб-сервис, который позволяет централизованно хранить, учитывать и обслуживать пул собственных Steam-аккаунтов через единый веб-интерфейс и API.

Сервис должен устанавливаться на сервер пользователя, не зависеть от сторонних SaaS-панелей и поддерживать пакетные операции по аккаунтам.

## 2. Границы проекта
### Входит в MVP
- учет и каталог Steam-аккаунтов;
- безопасное хранение учетных данных и служебных секретов;
- группировка аккаунтов по папкам, тегам и статусам;
- хранение заметок и метаданных по аккаунтам;
- проверка доступности аккаунта и базового состояния авторизации;
- запуск массовых задач по выбранным аккаунтам;
- изменение публичных параметров профиля там, где это технически возможно;
- аудит действий и журнал ошибок;
- self-hosted развертывание через Docker Compose;
- REST API и веб-интерфейс.

### Не входит в MVP
- торговые операции, маркет, обмены, инвентарь, продажа предметов;
- автоматизация, нарушающая правила площадки;
- массовая рассылка сообщений;
- антикапча, антибан, обход ограничений платформы;
- управление чужими аккаунтами без их согласия;
- обход Steam Guard или несанкционированный доступ.

## 3. Основные сценарии использования
1. Администратор добавляет аккаунт в систему.
2. Сервис сохраняет логин, пароль, служебные токены/секреты в защищенном хранилище.
3. Администратор назначает теги, папку, владельца, статус и заметки.
4. Администратор выбирает несколько аккаунтов и запускает пакетную операцию:
   - обновить display name;
   - обновить summary/описание;
   - обновить privacy settings;
   - обновить custom URL при возможности;
   - обновить аватар;
   - проверить состояние входа;
   - переавторизовать аккаунт;
   - экспортировать отчет по состоянию.
5. Сервис помещает операцию в очередь, выполняет по лимитам и пишет результат по каждому аккаунту.
6. Администратор смотрит подробный лог: что было запрошено, что применилось, где возникла ошибка.

## 4. Роли пользователей
### 4.1 SuperAdmin
- управление системой;
- управление пользователями и ролями;
- доступ к секретам через разрешенные операции;
- настройка интеграций и шифрования.

### 4.2 Admin
- управление аккаунтами;
- запуск массовых задач;
- просмотр журналов;
- импорт/экспорт метаданных.

### 4.3 Operator
- просмотр списка аккаунтов;
- запуск только разрешенных операций;
- без прямого доступа к секретам.

### 4.4 Auditor
- доступ только на чтение;
- просмотр истории действий, отчетов и результатов задач.

## 5. Функциональные требования

## 5.1 Модуль каталогизации аккаунтов
Система должна позволять:
- создавать карточку аккаунта;
- хранить логин Steam Account Name;
- хранить текущий profile name/display name;
- хранить SteamID/SteamID64, если известен;
- хранить e-mail аккаунта;
- хранить телефон/признак наличия телефона;
- хранить теги;
- хранить группу/папку;
- хранить заметки;
- хранить технический статус:
  - active;
  - requires_relogin;
  - steam_guard_required;
  - limited;
  - locked;
  - error;
  - archived.

Для каждого аккаунта должны отображаться:
- дата создания записи;
- дата последней успешной операции;
- дата последней ошибки;
- дата последней проверки;
- кто последний изменял запись.

## 5.2 Безопасное хранение секретов
Сервис должен хранить отдельно:
- пароль аккаунта;
- refresh/access/cookie-подобные токены, если применимо;
- shared secret / identity secret / иные служебные секреты, если пользователь законно хранит их для своих аккаунтов;
- резервные данные восстановления только как зашифрованные заметки;
- пользовательские прокси/сетевые параметры, если они привязаны к аккаунту.

Требования:
- шифрование секретов на уровне приложения;
- master key вне БД;
- поддержка внешнего secret storage;
- невозможность показать пароль в интерфейсе без отдельного разрешения и события аудита;
- все чтения секретов должны логироваться.

## 5.3 Авторизация и сессии аккаунтов
Система должна поддерживать:
- проверку валидности сессии аккаунта;
- повторную авторизацию;
- загрузку и обновление cookie/session container;
- учет статусов Steam Guard;
- пометку аккаунта как требующего ручного действия.

Сервис не должен обещать автоматический обход любых дополнительных защитных сценариев.

## 5.4 Массовые операции по аккаунтам
Система должна поддерживать пакетные задания для одного, нескольких или всех аккаунтов, отфильтрованных по условию.

Минимальный набор batch-операций:
- change display name;
- change profile summary;
- change real name;
- change location fields;
- change privacy settings;
- change custom profile URL;
- upload/change avatar;
- validate login/session;
- refresh web session;
- assign tags;
- move to group;
- add internal note;
- export selected accounts to CSV/XLSX/JSON без секретов.

Для batch-операций обязательны:
- dry-run режим;
- лимит параллелизма;
- retry policy;
- отмена задачи;
- per-account result;
- итоговый отчет;
- идемпотентность там, где возможно.

## 5.5 Очереди и фоновые задачи
Система должна иметь отдельный механизм фонового выполнения задач:
- постановка задач в очередь;
- приоритизация;
- ограничение конкурентности;
- backoff при ошибках;
- dead-letter сценарий;
- хранение истории выполнения.

Задача должна состоять из:
- типа операции;
- списка аккаунтов;
- полезной нагрузки;
- параметров выполнения;
- автора запуска;
- статуса;
- счетчиков успехов и ошибок.

## 5.6 UI и поиск
Веб-интерфейс должен обеспечивать:
- таблицу аккаунтов;
- быстрый поиск;
- фильтры по тегам, статусам, папкам, владельцу, времени последней проверки;
- массовое выделение;
- карточку аккаунта;
- страницу задачи с прогрессом;
- просмотр логов;
- простую форму импорта;
- темную тему как минимум опционально.

## 5.7 Импорт и экспорт
Импорт:
- CSV;
- JSON;
- опционально XLSX.

Поля импорта:
- login;
- password;
- email;
- displayName;
- tags;
- folder;
- note;
- proxy;
- custom metadata.

Экспорт:
- без секретов по умолчанию;
- отдельный административный экспорт с секретами только при наличии роли и отдельного подтверждения.

## 5.8 Аудит и журналирование
Сервис обязан вести аудит:
- входов в систему;
- попыток просмотра секретов;
- изменений карточек аккаунтов;
- запусков batch-задач;
- результатов операций;
- системных ошибок;
- действий пользователей по ролям.

Аудит должен быть неизменяемым на уровне бизнес-логики.

## 8. Интеграционный слой Steam
### 8.1 Подход
Внутри C#-сервиса должен быть выделен адаптерный слой `ISteamAccountGateway`, скрывающий конкретную реализацию операций.

Пример интерфейсов:
- `AuthenticateAsync()`
- `ValidateSessionAsync()`
- `RefreshSessionAsync()`
- `GetProfileAsync()`
- `UpdateProfileAsync()`
- `UpdateAvatarAsync()`
- `GetPrivacySettingsAsync()`
- `UpdatePrivacySettingsAsync()`
- `ResolveSteamIdAsync()`

### 8.2 Ограничения интеграции
Слой интеграции должен учитывать, что:
- часть данных и операций может быть нестабильной;
- часть операций может требовать актуальной веб-сессии;
- часть операций может перестать работать после изменений со стороны платформы;
- логин аккаунта не является изменяемым;
- batch-режим требует строгих лимитов и очередей.

## 12. Требования к security design
1. Все секреты шифровать AES-GCM или эквивалентом через централизованный криптосервис.
2. Master key хранить вне PostgreSQL.
3. Поддержать ротацию ключей.
4. Ограничить доступ к raw-секретам только сервисному слою.
5. Не показывать секреты в логах, exception message и telemetry.
6. Сессии пользователей сервиса должны инвалидироваться после смены критичных настроек.
7. Любой экспорт с секретами должен требовать отдельного подтверждения и записываться в аудит.
8. Желательно поддержать IP allowlist для админской панели.

## 13. Ограничения и честные допущения
- Не все пользовательские действия Steam гарантированно доступны через стабильный официальный интерфейс.
- Некоторые операции могут требовать обновления интеграционного адаптера после изменений на стороне платформы.
- Сервис не должен строиться на предположении, что любой аккаунт всегда можно обслуживать без ручного подтверждения.
- Логин Steam Account Name менять нельзя; допускается изменение только публичных данных профиля, если операция поддерживается.

## 17. Критерии приемки MVP
MVP считается принятым, если:
1. Можно поднять систему локально или на VPS одной командой через Docker Compose.
2. Можно создать пользователя-администратора.
3. Можно импортировать не менее 100 аккаунтов через CSV.
4. Можно просматривать, фильтровать и группировать аккаунты.
5. Можно безопасно хранить секреты.
6. Можно запускать batch-задачи минимум на 50 аккаунтов.
7. Можно получить детальный результат по каждому аккаунту.
8. Есть аудит действий.
9. Есть экспорт отчетов без секретов.
10. Есть документация по деплою и эксплуатации.

## 18. Дополнительные рекомендации по реализации
- Делать интеграцию со Steam через отдельный адаптерный слой, чтобы можно было менять реализацию без переписывания бизнес-логики.
- Не зашивать логику секретов прямо в сущности аккаунта.
- Все долгие операции вынести в фон.
- Сразу проектировать систему под частичные ошибки: часть аккаунтов пройдет, часть нет.
- Любую потенциально рискованную операцию сначала выполнять в dry-run.
- Для UI сразу сделать bulk-first UX: фильтр → выделение → шаблон операции → запуск → прогресс → отчет.

## 19. Желаемый итоговый результат
На выходе должен получиться self-hosted сервис, который:
- разворачивается у пользователя на своем сервере;
- позволяет централизованно вести пул собственных Steam-аккаунтов;
- безопасно хранит чувствительные данные;
- поддерживает массовые операции по профилям и сессиям;
- имеет web UI, REST API, аудит, очередь задач и пригоден для дальнейшего расширения.
